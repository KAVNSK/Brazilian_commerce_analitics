# Brazilian Commerce Analytics (Olist)

Это демонстрационный аналитический проект на датасете Olist.  
Бизнес-кейс: понять, почему часть заказов получает 1-2 звезды, и на какие зоны риска бизнесу смотреть в первую очередь.

Пайплайн:

- постановка бизнес-вопроса;
- сбор витрин в SQL (DuckDB);
- анализ качества данных;
- расчет KPI;
- визуализацию результатов;
- выводы в формате рекомендаций для бизнеса.

Рабочие гипотезы:
- задержки доставки увеличивают долю негативных отзывов;
- часть продавцов системно создает больше риска по сервису.
- есть группа продавцов, которая дает непропорционально много задержавшихся заказов с плохой оценкой.

Ключевые метрики:

- `bad_review_rate` - доля заказов с `review_score <= 2`;
- `late_rate` - доля заказов, доставленных позже обещанного срока;
- `delay_days` / `delivery_days` - разница между фактической и обещанной датой доставки.

Источник данных: [Brazilian E-Commerce Public Dataset by Olist](https://www.kaggle.com/datasets/olistbr/brazilian-ecommerce).

## Хронология работы (план -> реализация)

Ниже последовательность, по которой собран кейс. Это тот же план из `olist_analysis_plan.ipynb`, но уже вшитый в основной README.

### 1) Минимальная структура данных

Для кейса достаточно 5 таблиц:

| Файл | Зачем нужен |
|------|-------------|
| `olist_orders_dataset.csv` | даты и статус заказа |
| `olist_order_items_dataset.csv` | цена, фрахт и продавец |
| `olist_order_payments_dataset.csv` | тип оплаты и рассрочка |
| `olist_order_reviews_dataset.csv` | оценка клиента |
| `olist_customers_dataset.csv` | штат клиента |

Базовые связи:
- `orders` -> `order_items`, `payments`, `reviews` по `order_id`;
- `orders` -> `customers` по `customer_id`.

### 2) Что считаем в первую очередь

Обязательные метрики:
- `bad_review_rate`
- `late_rate`
- `delay_days`

Метрики для контекста:
- `orders`
- `GMV`
- `AOV`

Базовая статистика:
- хи-квадрат для связи `is_late` и `bad_review_flag`.

### 3) Порядок анализа

1. Загрузка данных в DuckDB.
2. Проверка качества данных (пропуски дат, orphan-строки, статусы).
3. Сбор двух витрин:
   - `v_order_facts` (общая картина),
   - `v_case_order_base` (late/bad/seller).
4. Расчет ключевых метрик.
5. Визуализация и проверка гипотез.
6. Формулировка действий для бизнеса.

### 4) Минимальный набор графиков

1. Динамика заказов и GMV по месяцам.
2. Распределение `delivery_days`.
3. Доля плохих отзывов по группам задержки.
4. Scatter продавцов: `late_rate` vs `bad_review_rate`.
5. Распределение `review_score`.

Этого набора хватает, чтобы связно защитить кейс без перегрузки.

## Что происходит в каждом ноутбуке

### `olist_analysis_plan.ipynb`

Это краткая дорожная карта проекта. Основные пункты из него перенесены в раздел "Хронология работы" в этом README.

### `olist_ecommerce_report.ipynb`

Это широкий обзор датасета и общей операционной картины.

Основные шаги:

1. Подключение библиотек и настройка среды.
2. Загрузка CSV в локальную DuckDB.
3. Проверка качества и целостности данных.
4. Сбор витрины `v_order_facts`.
5. Расчет базовых KPI (заказы, GMV, AOV, доля late-доставок, отзывы).
6. SQL-анализ динамики и концентрации продавцов.
7. Построение основных графиков.
8. Простые статистические проверки (Spearman, Mann-Whitney).
9. Черновые выводы.

Используемые методы:

- SQL (`JOIN`, `GROUP BY`, CTE, оконные функции);
- Python (`pandas`, `numpy`);
- визуализация (`matplotlib`, `seaborn`);
- статистика (`scipy.stats`).

### `olist_case1_customer_satisfaction.ipynb`

Это отдельный целевой кейс с понятным вопросом для бизнеса.

Что делается:

1. Формулируется цель и KPI.
2. Загружаются данные в DuckDB.
3. Строится витрина `v_case_order_base` на уровне заказа.
4. Заказы сегментируются по задержкам.
5. Сравнивается доля плохих отзывов между сегментами.
6. Считаются метрики продавцов и выделяется risk-группа.
7. Проверяется статистическая связь (хи-квадрат).
8. Формулируются действия для Customer Success / Operations.

Используемые методы:

- SQL-витрины и агрегации;
- pandas-анализ;
- bar/scatter визуализации;
- `chi2_contingency`.

### `olist_case1_solution_report.ipynb`

Это основной интегрированный отчет по кейсу.

Что внутри:

1. Формулировка кейса и гипотез.
2. Реляционный слой в DuckDB.
3. Первичная проверка качества данных.
4. Витрина заказов `v_order_facts` и обзорные метрики.
5. Целевая витрина `v_case_order_base`.
6. Анализ risk-продавцов и их вклада в `late+bad`.
7. Матстат-проверки (`chi2`, Spearman, Mann-Whitney).
8. Две дополнительные гипотезы (`H4`, `H5`) и быстрый скрининг.
9. Простой план проверки причинности (A/B или до/после + контроль).

## Ключевые выводы по кейсу

По текущему прогону:

- доля плохих отзывов при late-доставке: **53.99%**;
- доля плохих отзывов при on-time/early: **9.19%**;
- эффект почти **5.9x** в пользу гипотезы о влиянии просрочки;
- продавцов с `orders >= 50`: **413**, из них в risk-группе: **114**;
- хи-квадрат, Spearman и Mann-Whitney подтверждают устойчивую статистическую связь.

Если коротко: логистика и SLA напрямую связаны с клиентской оценкой.  
А еще видно, с кого начинать операционные улучшения — это группа risk-продавцов.

## Дополнительные гипотезы

После основного анализа добавлены гипотезы следующего шага:

- `H4`: в штатах с более высоким `late_rate` выше `bad_review_rate`;
- `H5`: большее число платежей/рассрочка связано с более высоким риском негатива;

Обе гипотезы уже добавлены в `olist_case1_solution_report.ipynb` как скрининговый блок с графиками.

## Как проверять причинность

В проект добавлен практический план проверки причинности:

- **A/B пилот** на risk-продавцах:
  - test-группа с операционными мерами;
  - control-группа без изменений;
  - главная метрика: `late_bad_rate`.
- **Если A/B нельзя запустить**:
  - сравнение до/после;
  - плюс контрольная группа похожих продавцов.

Такой подход помогает перейти от "похоже, что связано" к более надежной проверке эффекта.

## Как запустить локально

1. Установить зависимости:
   - `pip install -r requirements.txt`
2. Проверить, что папка `data/` содержит CSV Olist.
3. Запустить Jupyter:
   - `jupyter notebook`
4. Открывать ноутбуки в порядке:
   - `olist_analysis_plan.ipynb` (короткий roadmap)
   - `olist_ecommerce_report.ipynb`
   - `olist_case1_customer_satisfaction.ipynb`
   - `olist_case1_solution_report.ipynb`

## Что важно учитывать

- Источник данных: Kaggle, `olistbr/brazilian-ecommerce`.
- Проект учебно-демонстрационный, не production-сервис.
- Некоторые результаты зависят от выбранных допущений (например, как назначается `primary_seller`).
